# Skill: JSONL 题单讲题专题生成器

## 触发条件

当用户提供一个 `jsonl` 题单，并要求：
- 根据题单重写 / 生成专题文档
- 参考某篇现有 `immerse` 文档的 style 来“讲题”
- 明确要求“只按 jsonl 的题目和链接生成”

使用本 skill。

如果用户同时提供：
- 一个 `jsonl` 文件
- 一个旧的目标 `.md`
- 一个参考风格 `.md`

则遵循以下优先级：

1. **`jsonl`**：题目与链接的唯一事实来源
2. **参考风格 `.md`**：只参考版式、章节层级、语气节奏
3. **目标 `.md`**：默认整篇重写；其旧内容不得作为题目或正文来源

---

## 输入约定

`jsonl` 每行一个 JSON 对象。优先读取以下字段：

| 信息 | 优先字段 |
| --- | --- |
| 题目标题 | `luogu_problem_title` → `title` → `problem_title` |
| 题目链接 | `luogu_url` → `url` → `link` |
| 标签 | `tags` |
| 题意摘要 | `luogu_problem_description` → `description` |

若字段名不完全一致，按“标题 / 链接 / 标签 / 描述”的语义做兼容映射。

---

## 工作流程

### Phase 1: 读取题单

1. 逐行解析 `jsonl`
2. 提取题目标题、链接、标签、题意摘要
3. 按 **链接去重**，保留第一次出现的顺序
4. 若用户明确指出某些题是“重复题 / 同题复现 / 同模型镜像”，允许在按链接去重之后继续删去这些**明显重复**的条目，并以保留后的顺序为准重新编号
5. 统计最终题目数，后续所有编号和总数都以此为准

### Phase 2: 理解专题

1. 从 `jsonl` 文件名、所在目录、用户描述中识别专题方向
2. 可以用 `web_search` 补充题意理解或常见做法
3. **不得**新增题单外题目
4. **不得**替换题单中的题目标题与链接

### Phase 3: 重组章节

1. 不机械照抄 `jsonl` 顺序
2. 按知识脉络将题目重组为 **3-6 个大章**
3. 章节标题采用“讲题式专题”风格，例如：
   - `# 一、DFS 入门与基础搜索`
   - `# 二、BFS 与状态图搜索`
   - `# 三、启发式搜索与迭代加深`
4. 每章开头写 **1 小段短导语**（通常 1-3 句），说明本章题型范围与共同方法
5. 若某些题与当前章节标题明显不匹配，应调整章节分组或章节命名；不要为了凑数把题硬塞进不合适的桶

### Phase 4: 生成文档

1. 产物是 **讲题式专题文档**
2. frontmatter 后只保留：
     - 文档标题
     - 1 段优先、必要时 2 段的简短总述（只讲专题范围、题型脉络、阅读方式）
     - 直接进入分章讲题
3. 默认**不写**：
     - “什么是该算法”
     - “核心思想总论”
    - “技巧总结”
     - “路线图 / 总览表 / 口诀”
     - “按照 xxx.jsonl 整理”
     - “原始记录共 X 条，去重后共 Y 道”
     - “正文只按题单重写，旧文不作为正文来源”
     - “本专题收录 N 道题”
     - “按 A → B → C 展开”这类偏目录说明、缺少主题感的开头
4. 只有用户明确要求时，才追加这些综述章节

### Phase 5: 验证

1. `## N. ...` 题目标题数量 = 最终保留的题目数
2. 题目编号必须连续
3. 最终保留的每个链接都必须出现在正文中
4. 行尾统一为 **CRLF / `\r\n`**

## 仓库特定风格约束（algorithm-docs / immerse）

### 标题

1. `title` 与正文 `H1` 必须完全一致
2. 标题统一用：
   - `{来源} {高层主题}专题精选解题报告`
3. 其中：
   - `{来源}` 常见为 `洛谷` / `Codeforces` / `AtCoder` / `LeetCode` / `HDU` / `USACO` 等
   - `{高层主题}` 必须是**高层、可归类的主题名**
4. 若标题开始变成长串并列名词（例如“数论、矩阵与高斯消元”“几何、博弈与异或工具”），优先**压缩为更高层的大类**，把细分方向放到 `subtitle` 与开头总述里
5. 标题禁止出现题目数量

### subtitle

1. `subtitle` 负责展开专题跨度与细分方向
2. `subtitle` 默认写成“从 A 到 B ...”的主题弧线
3. `subtitle` 禁止出现：
   - `N 道`
   - `题目数量`
   - 统计口径说明

### 开头总述

1. 默认参考 `docs/immerse/math/luogu-geometry-game.md` 的 opening 风格
2. 开头应像：
   - “这一组题从 A 一路走到 B，题型跨度很大，但核心都在 ...”
3. 开头重点是：
   - 题型演进
   - 共同建模核心
   - 阅读主线
4. 开头禁止写成：
   - “本专题收录 N 道题”
   - “按 A → B → C 展开，统一整理为 ...”
   - 题单来源、去重数量、旧文处理方式

### 分章

1. 一级章必须使用 `# 一、...`、`# 二、...`
2. 每章都要有一小段导语，不要只剩一个空标题
3. 章节划分必须服务于“方法边界清晰”，而不是只追求平均分配题数
4. 若用户要求重写已有专题文档，且发现原分类或题目归属明显不合理，应直接重组，不必拘泥于旧文的分桶

## 每题的固定格式

每道题必须严格保持以下顺序：

1. `## N. [题目标题](题目链接)`
2. 一行标签：`` `标签1` `标签2` ``
3. `### 题意`
4. `### 分析`
5. `### 核心代码`
6. `### 复杂度`

默认**不加**：
- `核心启示`
- `答案位置`
- `扩展题`
- `题外话`

除非用户明确要求。

默认写法再补充两条：

- `题意` 不只压成一句话；优先把对象、限制、目标补全到可直接进入建模的程度
- `分析` 不写成纯结论摘要，尽量像讲题一样交代“怎么想到这个状态 / 为什么是这个窗口或凸包 / 最后答案取哪里”

## 代码风格

### 总原则

- 只给**核心代码**
- 代码必须与题意、分析保持一致
- 重点是“题解骨架”，不是完整可运行程序

### 具体规则

- 不写 `#include`
- 不写 `using namespace std;`
- 不写 `int main()`
- 保留问题特有的全局变量、结构体、核心函数
- 代码长度通常控制在 **5-25 行**
- 命名简洁：`n`、`m`、`g`、`vis`、`dist`、`dfs`、`bfs`、`ans`
- 不保留未使用变量
- 注释只解释关键语句，不写讲课式废话
- 若答案不等于最后一个状态，核心代码应补出答案提取部分，保证和分析对齐
- 代码块内部保持紧凑，**不要**在每一行代码之间额外插空行

---

## 排版风格

1. **全文中文**
2. 一级章节使用 `# 一、...`、`# 二、...`
3. 题目使用 `## 1. ...`、`## 2. ...`
4. 题与题之间用 `---`
5. 标签用单独一行的行内代码
6. 代码块必须使用三反引号，C++ 代码用 ` ```cpp `
7. 正文保持紧凑，不要出现“每行之间再空一行”的松散排版；尤其代码块内严禁隔行空行
8. 风格参考用户指定的文档；若用户未指定，则默认靠近 `docs/immerse/graph/luogu-graph-theory.md` 这种“直接讲题”的节奏

---

## 内容原则

1. **题单驱动**：题目来自 `jsonl`，不是自己扩题
2. **风格复刻**：参考文档只学版式和节奏，不借用其题目与正文
3. **直接讲题**：简短引入后立刻进入题目讲解
4. **按知识脉络重排**：章节服务于阅读体验，而不是被 `jsonl` 原顺序绑死
5. **允许补标签**：如果题单标签缺失，可根据做法补最简洁的标签
6. **默认整篇重写**：用户说“按 jsonl 重写”时，不在旧文上修补
7. **可按用户要求删明显重复题**：若用户明确要求去掉同题复现或明显重复条目，应直接删去并顺延后续编号
8. **计数严格一致**：最终保留的题目总数、标题编号、正文覆盖必须一致
9. **分析偏讲题式**：相比结论式摘要，更强调状态从哪来、优化为什么成立、答案最后落在哪里
10. **代码与讲解同步**：代码骨架应体现分析里提到的关键状态、转移与答案提取
11. **前言去元信息**：除非用户明确要求，正文开头不要写题单来源、去重数量、旧文处理方式等过程性说明
12. **标题尽量短**：标题负责归类，细分方向留给 `subtitle` 和 opening，不要把标题写成并列术语清单
13. **title / H1 同步**：修改标题时必须同时修改 `title` 与正文 `H1`
