# Skill: 算法专题教程生成器

## 触发条件
当用户要求生成某个算法专题的总结、教程、学习笔记、tutorial 时，使用本 skill。
如果工作区里已有同类教程，但其结构、代码风格或排版与本 skill 冲突，**一律以本 skill 为准**，不要把那类文件当作风格参考。

---

## 工作流程

### Phase 1: 调研与规划
1. **识别算法类型**：判定属于哪个大类（DP、贪心、搜索、二分、模拟、分治、数据结构、数学、构造等），后续分析模板据此选择
2. 使用 `web_search` 搜索该算法的经典题目、核心模型、解题模板
3. 收集 **15-20 道经典例题**，按难度分 4-5 级
4. 确定该算法有哪几种子类型/变体（如 DP 的背包/区间/树形，排序的归并/快排/桶排，搜索的 BFS/DFS/A*）

### Phase 2: 生成文档
文件名：`{N}.{topic}-tutorial.md`（N 为排序序号，如 `1.simulation-tutorial.md`）
保存位置：`docs/{分类}/` 目录下（如 `docs/dp/`、`docs/basic/`、`docs/search/`）
**行尾格式：必须使用 \r\n（Windows 换行）**

### Phase 3: 扩展已有教程（当用户要求在现有教程中追加题目时）
1. 读取现有文件，统计当前题目数和难度分布
2. 确定需要追加的题目及其所属难度级别
3. 在对应难度级别末尾插入新题，**重新编号**使全文题目连续（题目 1、2、3...）
4. 同步更新：`## 四、N 道` 标题中的 N、`### 7.3` 路线图、`### 7.4` 总览表
5. 验证：CRLF 换行、三反引号代码块、题目编号连续、计数一致

## 文档标准结构

```markdown
---

order: {N}
---

## 二、核心思想与前置知识
## 三、分类与变体详解（视算法而定）

## 五、常见技巧与思路总结
## 六、优化方法 / 解题思路框架
## 七、终极总结与归纳
```

### YAML Frontmatter 说明
- `title`：文章标题，不需要在正文写 `# 标题`
- `subtitle`：带 emoji 的副标题，不需要在正文写 `## emoji 副标题`
- `order`：排序序号，用于文档列表排序
- **不要**在正文中写手动目录（`## 目录` + 链接列表），TOC 由服务端自动生成
- frontmatter 结束后正文直接从 `## 一、...` 开始，中间不加 `---` 分隔线

### 各章节规范

**一、什么是{算法名称}**
- 2-3 段通俗解释
- ASCII art 图示（可选）
- 与相关算法的对比表格

**二、核心思想与前置知识**
- `### 2.1` / `### 2.2` / `### 2.3` 子节编号
- 2.x 通用框架代码：默认使用 **关键变量 + 关键函数 + 核心算法骨架** 的讲义风格，不写完整程序
- 仅当用户**明确要求完整模板/完整可运行程序**时，才允许 `#include`、`main()`
- 即使是骨架代码，也必须保留关键初始化、可达性判断、合法性判断和边界处理，不能省略到会误导读者
- 骨架代码里不得保留未使用变量
- 其余子节用文字 + 代码块 + 表格

**三、分类与变体详解**
- `### 3.1` / `### 3.2` 等
- 每个变体/子类型：简要说明 → 核心要素摘要（代码块，格式见下方“算法类型分析模板”）→ 简洁模板代码（```cpp```）
- 特殊实现细节（如 DP 的 bitmask 映射、搜索的状态编码）须显式说明
- 结尾可加变体对比总表

## 代码风格（最关键！）

### 总原则
- **正确性优先于篇幅、排版和炫技**
- 任何“简化骨架”的行为，都不能省掉关键 base case、不可达态处理、越界 guard、合法性判断
- 文中分析块（状态定义/贪心策略/check 条件等）必须与代码骨架一致，不能前后口径不同

### 例题代码 — 简洁风格
- **禁止**：`#include`、`using namespace std;`、`int main()`、`scanf`/`cin`、`return 0`
- **保留**：
  - 问题特有的全局变量：`const int MAXN = 305;`、`int dp[MAXN][MAXN];`、`int ans;`、`vector<int> adj[MAXN];`
  - 必要的结构体：`struct Edge { int to, w; };`、`struct Node { ... };`
  - 核心函数（`void dfs(...)` / `void solve(...)` / `bool check(...)` / 直接循环体）
- **注释职责**：只解释变量、辅助函数或关键语句的作用；不要写“答案：...”这类讲课提示
- **答案位置**：统一放在代码块外的“答案位置：...”一行说明，不在代码注释里重复
- 代码长度：通常 5-25 行
- 变量命名简洁：`sz`、`adj`、`dp`、`f`、`ans`
- 命名尽量前后一致：如 `states/st`、`id/pre_id`、`compat`、`last_offset`
- 禁止保留无意义或未使用变量（如教程骨架里定义了 `bits` 却完全不用）

### 模板/框架代码 — 精简骨架为主（section 二 默认）
- 默认不给完整程序，只给关键变量、关键函数与核心算法骨架
- 若用户明确要求完整模板，才允许 `#include <bits/stdc++.h>`、`using namespace std;`、`int main()`
- 即使给骨架，也要用 best practice：变量有意义、初始化清晰、guard 明确、风格统一

### 示例对比
```
❌ 错误（例题中不应出现）:
#include <bits/stdc++.h>
using namespace std;
int dp[305][305];
int main() {
    int n; cin >> n;
    ...
    cout << dp[1][n] << endl;
}

✅ 正确（例题代码风格）:
const int MAXN = 305;
int dp[MAXN][MAXN];

void solve(int n) {
    for (int len = 2; len <= n; len++) {
        for (int i = 1; i + len - 1 <= n; i++) {
            int j = i + len - 1;
            dp[i][j] = 0x3f3f3f3f;
            for (int k = i; k < j; k++){
                dp[i][j] = min(dp[i][j], dp[i][k] + dp[k+1][j] + pre[j] - pre[i-1]);
            }
        }
    }
}
// pre[i] = a[1]+...+a[i]; solve(n)
答案位置：dp[1][n]
```

## 排版风格

1. **全文中文**，代码/算法名可用英文
2. **章节编号**：一级用中文数字（一、二、三），二级用阿拉伯数字（2.1、3.2）
3. **分隔线**：题目之间用 `---`，难度级别只用单行 `### 🟢 入门级 Level 1：xxx`（不加 ═══ 装饰线）
4. **代码块**：必须使用三个反引号（` ``` `），状态定义用 ` ``` `（无语言标记），C++ 代码用 ` ```cpp `。**严禁使用单反引号作为代码块围栏**
5. **强调**：关键词用 `**加粗**`，核心公式用行内代码 `` `dp[i][j]` ``
6. **表格**：对齐，用于对比、总览、信号词映射等

## 内容原则

1. **渐进式**：从最简单到最复杂，后题复用前题思维
2. **正确性第一**：文字解释、分析块和代码骨架必须一致，不能为了简洁牺牲准确性
3. **best practice**：骨架代码也应具备良好命名、无未使用变量、关键 guard 明确
4. **本 skill 优先**：若本地已有教程与本 skill 冲突，以本 skill 为最终标准
5. **核心启示要可迁移**：不是就题论题，而是提炼通用方法论
6. **图解辅助**：ASCII art 展示算法过程（状态转移、搜索树、分治递归等）
7. **口诀收尾**：帮助快速回忆整个知识体系，4-8 行押韵中文
8. **题目来源多样**：混合 LeetCode、洛谷、AcWing、CSES、经典竞赛题
9. **每题独立可读**：不依赖前文代码，每题的分析块和代码自成一体
