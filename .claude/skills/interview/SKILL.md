# Skill: 面试专题教程生成器

## 触发条件
当用户要求生成面试相关的算法/数据结构专题教程时，使用本 skill。
面试专题与算法竞赛专题（algorithm skill）的区别：
- **面试专题**：场景驱动、难度递进、重设计思路和方案对比、代码精简只给核心
- **算法竞赛专题**：题型分类、例题详解、完整模板代码、口诀总结

如果工作区里已有同类教程，但其结构、代码风格或排版与本 skill 冲突，**一律以本 skill 为准**。

---

## 工作流程

### Phase 1: 调研与规划
1. **识别专题方向**：判定属于哪个面试大类（数据结构设计、算法策略、系统设计等）
2. 使用 `web_search` 搜索该专题的高频面试题、经典场景、方案对比
3. 收集 **5-8 个核心知识点**，按难度分 3-5 级（⭐ 到 ⭐⭐⭐⭐）

### Phase 2: 生成文档
文件名：`{N}.{topic}-tutorial.md`（N 为排序序号）
保存位置：`docs/immerse/interview/` 目录下
**行尾格式：必须使用 \r\n（Windows 换行）**

### Phase 3: 扩展已有教程（当用户要求追加内容时）
1. 读取现有文件，理解当前结构和难度分布
2. 在对应难度级别中插入新内容
3. 同步更新：概述表格、总结章节的选型指南和速查表
4. 验证：CRLF 换行、代码自包含、难度标注一致

## 文档标准结构

```markdown
---
title: "{面试专题名称}"
subtitle: "{emoji} {一句话定位}"
order: {N}
---
## 一、概述
## 二、{最简单的知识点} ⭐
## 三、{中等知识点} ⭐⭐
## 四、{中等知识点} ⭐⭐
## 五、{较难知识点} ⭐⭐⭐
## 六、{最难知识点} ⭐⭐⭐⭐
## 七、总结
```

### YAML Frontmatter 说明
- `title`：面试专题名称，简洁直接
- `subtitle`：带 emoji 的一句话定位
- `order`：排序序号
- **不要**在正文中写手动目录，TOC 由服务端自动生成
- frontmatter 结束后正文直接从 `## 一、...` 开始

### 各章节规范

**一、概述**
- 1-2 段点明本专题在面试中的定位
- 一个**难度总览表**：列出所有知识点、典型场景、核心复杂度、难度星级
- 不要长篇大论，快速进入正题

**二 ~ 六、各知识点章节**（按难度从低到高排列）

每个知识点章节遵循统一结构：

```
### N.1 场景
  用 1-2 个真实面试场景引入：
  - 用加粗标出场景名
  - 用代码块做需求拆解（暴力方案 → 优化思路 → 为什么选这个结构）

### N.2 核心实现
  - 只给核心代码，不要完整程序
  - 代码必须自包含（不引用其他章节未定义的函数/类）
  - 可以提供多种写法（标注"写法一"、"写法二"），但每种都要完整可用
  - 代码后紧跟复杂度表格（操作 | 复杂度 | 说明）

### N.3 面试实战
  - 每个实战题用加粗标题，标注 LeetCode 题号
  - 给出核心代码（5-15 行）
  - 方案对比用表格（方案 | 复杂度 | 适用场景）
  - 面试追问用列表或代码块

### N.4 面试延伸（可选）
  - 变体题目、系统设计追问、"不能做什么"的边界讨论
```

**七、总结**

```
### 7.1 选型指南（代码块，问自己的问题 → 对应结构）
### 7.2 核心设计原则（3-5 条编号列表）
### 7.3 面试速查表
  | 面试问题 | 数据结构 | 关键思路 |
```

## 代码风格（最关键！）

### 总原则
- **精简核心，不要完整程序**
- 代码量控制在面试白板可手写的范围（5-25 行/段）
- 代码必须**自包含**：一个代码块内用到的类/函数必须在同一个代码块内定义，或在前面章节已完整给出
- 用最简单的类型（`int` 优先），不要过度模板化

### 具体规则
- 不写 `#include` / `using namespace std;` / `int main()`
- 类和函数直接给定义体
- 变量命名简洁：`fa`、`dq`、`mp`、`cnt`
- 注释只标注关键设计决策，不要逐行解释
- 一个概念有多种写法时（如 MinStack 的 pair 写法 vs 双栈写法），都要给出，让读者选择
- 面试实战的代码可以引用前面定义的类（如 `Heap h;`），但被引用的类必须在前面完整给出过

### 示例
```
❌ 错误（太重，面试写不完）：
template <typename T, typename Compare = greater<T>>
class Heap {
    vector<T> data;
    Compare comp;
    ...
};

✅ 正确（精简核心）：
class Heap {
    vector<int> data;
    void up(int i) {
        int p = (i-1)/2;
        if (i > 0 && data[i] < data[p]) { swap(data[i], data[p]); up(p); }
    }
    void down(int i) {
        int best = i, l = 2*i+1, r = 2*i+2, n = data.size();
        if (l < n && data[l] < data[best]) best = l;
        if (r < n && data[r] < data[best]) best = r;
        if (best != i) { swap(data[i], data[best]); down(best); }
    }
public:
    void push(int v) { data.push_back(v); up(data.size()-1); }
    void pop() { swap(data[0], data.back()); data.pop_back(); down(0); }
    int top() { return data[0]; }
    int size() { return data.size(); }
    bool empty() { return data.empty(); }
};
```

## 排版风格

1. **全文中文**，代码/算法名/LeetCode 题号用英文
2. **章节编号**：一级用中文数字（一、二、三），二级用阿拉伯数字（2.1、3.2）
3. **难度标注**：章节标题末尾用 ⭐ 标注难度（⭐ 到 ⭐⭐⭐⭐）
4. **分隔线**：章节之间用 `---`
5. **代码块**：必须使用三反引号，C++ 代码用 ` ```cpp `，伪代码/需求分析用 ` ``` `
6. **强调**：关键词用 `**加粗**`
7. **表格**：用于对比、总览、复杂度分析
8. **图解**：用 ASCII art 展示数据流动、树形结构、链表操作等
9. **面试追问**：用加粗问题 + `→` 简短回答的格式

## 内容原则

1. **场景驱动**：每个知识点从真实面试场景切入，不要上来就讲定义
2. **难度递进**：全文从 ⭐ 到 ⭐⭐⭐⭐，后面的知识点可以复用前面的结构
3. **方案对比**：同一个问题给出多种解法，用表格对比适用场景
4. **代码自包含**：读者看任何一个代码块都不需要翻前面找缺失的定义
5. **面试视角**：每个知识点都要回答"面试官会怎么追问"
6. **善用 STL**：面试中用 STL 不丢分，理解原理 + 灵活使用才是加分项
7. **精简优先**：宁可少讲一个知识点，也不要把一个知识点讲得又臭又长
8. **引用用户代码**：如果用户有已有实现，优先基于用户代码改写，保持风格一致
